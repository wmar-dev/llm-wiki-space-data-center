## 1. Playwright MCP Configuration

**Decision**: Use the `@playwright/mcp` MCP server, configured in Claude Code's
`.claude/mcp.json` (or `~/.claude/mcp.json` for global config).

**Configuration** (`.claude/mcp.json`):

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}
```

**Key tools exposed**: `browser_navigate`, `browser_snapshot`, `browser_click`,
`browser_type`, `browser_pdf_save`. The `browser_navigate` + `browser_snapshot`
combination covers the primary fetch-and-extract workflow (FR-011, FR-015).

**Rationale**: Playwright MCP is the user-specified tool. It handles JS-heavy pages
that basic HTTP cannot render, which is the primary value over curl/wget.

**Failure fallback** (FR-016): On Playwright failure, retry once; then fall back to
`curl -L` for basic HTTP fetch. Log both attempts with outcome.

**Alternatives considered**: `fetch` MCP (simpler but no JS rendering), curl (no JS).
Neither meets the spec requirement for full page rendering.

## 2. PDF Extraction

**Decision**: Use `pdftotext` (from `poppler-utils`) as the primary CLI tool.
Fall back to Python `pdfplumber` if `pdftotext` is unavailable.

**Installation**:

```bash
brew install poppler          # macOS — provides pdftotext
pip install pdfplumber        # Python fallback
```

**Wrapper** (`tools/pdf-extract.sh`): Detects available tool, extracts to stdout,
saves alongside original as `<name>.txt` in `raw/`.

**Rationale**: `pdftotext` is fast, zero-Python dependency, and handles the vast
majority of text-layer PDFs. `pdfplumber` handles complex layouts better but requires
Python. Image-only PDFs are out of scope (per spec Assumptions).

**Alternatives considered**: Playwright `browser_pdf_save` (renders visually, doesn't
extract text layer), LLM vision pass (expensive, violates Principle VII).

## 3. Marp Slide Generation

**Decision**: Use `@marp-team/marp-cli` installed globally via npm.

**Installation**:

```bash
npm install -g @marp-team/marp-cli
```

**Usage pattern**: The query skill generates a `.md` file with Marp frontmatter
(`marp: true`, theme, size). The researcher runs `marp <file>.md` to produce HTML/PDF.
The LLM does not run Marp itself — it generates the input file; slide rendering is
a researcher step.

**Rationale**: Keeping Marp invocation in the researcher's hands avoids requiring
subprocess execution from Claude Code. The LLM's job is to produce valid Marp Markdown.

**Alternatives considered**: Pandoc + reveal.js (heavier), direct HTML generation
(not Marp-compatible for Obsidian plugin use).

## 4. Matplotlib Chart Generation

**Decision**: The query skill generates a self-contained Python script that produces
the chart. The researcher runs the script; the output PNG is filed into `wiki/`.

**Pattern**:

```python
# Generated by LLM — run with: python chart_<topic>.py
import matplotlib.pyplot as plt
# ... chart data and rendering ...
plt.savefig("wiki/comparisons/<topic>-chart.png")
```

**Rationale**: Claude Code can execute Python scripts via Bash tool. For cases where
the researcher wants the LLM to generate AND run the chart, the query skill can invoke
`python <script>` directly. Both modes are supported; the skill documents which was used.

**Alternatives considered**: Plotly (interactive but heavier), vega-lite (JSON spec,
less familiar to LLM). Matplotlib is the spec-specified tool.

## 5. Search Tooling Upgrade Path

**Decision**: At ≤200 sources, `index.md` + `grep`/`ripgrep` is sufficient.
Beyond 200 sources, introduce `qmd` (local hybrid BM25/vector search for Markdown).

**Current approach** (SC-006 compliant):

```bash
# tools/search.sh — ripgrep-based, returns top matches with file paths
rg --type md -l "$1" wiki/
```

**Upgrade trigger**: When `tools/search.sh` returns >10 results for routine queries,
or `index.md` exceeds ~500 lines. At that point propose `qmd` installation per FR-009.

**qmd** (reference from llm-wiki.md): `github.com/tobi/qmd` — local BM25/vector
hybrid search with MCP server. Install: `brew install qmd` (or from source).

**Rationale**: Introducing qmd before it's needed violates constitution Principle VII
(don't build tools for anticipated rather than demonstrated problems). The upgrade path
is documented so the planner can act immediately when the threshold is reached.

**Alternatives considered**: Embedding-based RAG (requires external API or local
model), SQLite FTS (overkill for Markdown files), Obsidian search (UI-only, not
LLM-accessible).

## 6. Claude Skills Format

**Decision**: Claude skills are Markdown files in `.claude/skills/`. Each skill is
invoked when the researcher types `/<skill-name>` in Claude Code, or when another
skill calls `EXECUTE_COMMAND: <skill-name>`.

**Format**:

```markdown
# Skill Name

Brief description.

## When to use
...

## Steps
1. Step one
2. Step two
...
```

**Versioning**: Skills are versioned via Git. The skill file includes a
`**Version**: X.Y.Z` line in its header. When a failure mode is observed, the skill
is updated and the version incremented per constitution Principle VI.

**Rationale**: Skills encapsulate the multi-step ingest/query/lint workflows so the
researcher invokes a single command rather than re-describing the procedure each session.
This is the primary mechanism for constitution Principle VI skill generation compliance.

## 7. Self-Evaluation Tooling

**Decision**: Micro evaluation is appended to `wiki/meta/evaluations.md` at session
end by the lint or query skill. Macro evaluation is triggered manually or after every
10 ingest operations (tracked via log entry count).

**Log parsing** for macro trigger:

```bash
grep -c "^## \[" log.md   # Count total log entries
grep -c "ingest" log.md   # Count ingest operations
```

**Evaluation format** (`wiki/meta/evaluations.md`):

```markdown
## [YYYY-MM-DD] Evaluation — <trigger: session-end | macro-10-sources>

| Dimension | Score | Notes |
| --------- | ----- | ----- |
| Token Economy | On Target | Queries averaged 3 pages |
| Wiki Health | On Target | 0 unresolved contradictions |
...

**Highest-impact improvement**: <proposed action>
```

**Rationale**: Structured format enables trend detection across macro cycles. The
`## [YYYY-MM-DD]` prefix makes entries grep-parseable (per llm-wiki.md log convention).
